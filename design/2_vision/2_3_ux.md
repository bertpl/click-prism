# 2.3. UX

## 2.3.0. Preamble

What `click-prism` should look and feel like in practice — for the
developer writing code and the end user running the CLI. The
sub-documents below present concrete scenarios: developer code alongside
expected output, establishing the desired behavior that the design phase
(section 3) needs to deliver.

### 2.3.0.1. Capabilities

`click-prism` unlocks five capabilities:

| Capability | How | Audience |
|---|---|---|
| Visual tree in help | `projex --help` (tree-as-help) | End user |
| Full CLI structure as JSON | `projex --help-json` | Tooling, agents |
| Visual tree (Python) | `show_tree(cli)` | Developer |
| JSON export (Python) | `show_tree(cli, format="json")` | Developer |
| Tree subcommand with runtime controls | `cli.add_command(tree_command())` | Opt-in |

Both Python-side rows can alternatively use `render_tree(cli, ...)`
to return the rendered output as a string instead of writing to
stdout — same defaults, same format support.

The first two are automatic when using `cls=PrismGroup` on the root
group. The tree subcommand is not created automatically — it requires
an explicit `cli.add_command(tree_command())`, keeping the CLI's command
namespace clean. The developer adds it when they want end-user runtime
controls like `--depth`.

### 2.3.0.2. Sub-sections

Organized by integration context. Developer code and end-user output
are shown together in each scenario — they are two sides of the same
coin.

- [2.3.1. Plain Click](2_3_1_plain_click.md) — The core story: integration
  paths, runtime controls, configuration, tree-as-help, parameter
  display, machine-readable output, edge cases. Plain Click only — no
  `rich`, no third-party extensions.

- [2.3.2. Rich](2_3_2_rich.md) — What changes when `rich` is available:
  auto-detection, graceful degradation, theming, bold group rows as
  visual hierarchy cues.

- [2.3.3. Composability](2_3_3_composability.md) — Combining `click-prism`
  with the relevant neighbor packages from section 1.1.4: `rich-click`, `cloup`,
  `click-extra`, `click-help-colors`, `click-didyoumean`,
  `click-default-group`, `click-aliases`. Multi-plugin stacking and
  fallback paths.

- [2.3.4. Requirements](2_3_4_requirements.md) — Consolidated, numbered
  requirements (R1–R41) derived from all scenarios in sections 2.3.1–2.3.3,
  organized by concern: integration, tree output, tree subcommand,
  tree-as-help, configuration, `rich`, composability, robustness.
